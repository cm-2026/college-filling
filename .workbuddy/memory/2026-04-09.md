## 修改查询逻辑：使用score_rank_table表获取位次

**修改时间**: 2026-04-09

### 需求说明
1. **分数模式**：用户输入分数500、地区河南、选科物理化学生物
   - 从score_rank_table查询500分对应的位次
   - 从score_rank_table查询490分（500-10）对应的位次
   - 用这两个位次的范围查询admission_plan表

2. **位次模式**：用户输入位次、地区、选科
   - 从score_rank_table查询最接近的分数
   - 查询该分数-10分对应的位次
   - 用这两个位次的范围查询admission_plan表

### 技术实现

#### 1. 数据表结构
- **score_rank_table**：分数位次对照表
  - province: 省份
  - subject_type: 科目类型（物理/历史/综合）
  - score: 分数
  - cumulative_count: 累计人数（位次）

- **admission_plan**：招生计划表
  - min_rank_1: 专业最低位次
  - group_min_rank_1: 专业组最低位次
  - subject_type: 科目类型（物理/历史）

#### 2. 查询流程修改

**修改文件**: `e:\xm\backend\server.js`

**新增逻辑**（第485-633行）：
1. 根据选科确定subject_type_for_rank
   - 3+3模式（北京/天津/上海/山东/浙江/海南）：'综合'
   - 3+1+2模式：第一门（物理/历史）

2. 分数模式查询位次：
   ```javascript
   // 查询用户分数对应位次
   SELECT cumulative_count FROM score_rank_table 
   WHERE province = ? AND subject_type = ? AND score = ?
   
   // 查询分数-10对应位次
   SELECT cumulative_count FROM score_rank_table 
   WHERE province = ? AND subject_type = ? AND score = ?
   ```

3. 位次模式查询分数：
   ```javascript
   // 查询最接近的分数
   SELECT score FROM score_rank_table 
   WHERE province = ? AND subject_type = ?
   ORDER BY ABS(cumulative_count - ?)
   
   // 查询分数-10对应位次
   SELECT cumulative_count FROM score_rank_table 
   WHERE province = ? AND subject_type = ? AND score = ?
   ```

4. 使用位次范围查询admission_plan：
   ```sql
   WHERE COALESCE(group_min_rank_1, min_rank_1) >= minRank
   AND COALESCE(group_min_rank_1, min_rank_1) <= maxRank
   ```

#### 3. 验证结果

**测试案例1**：河南物理500分
- 500分位次：209695
- 490分位次：229094
- 查询结果：1736条数据

**测试案例2**：河南物理位次10000
- 最接近分数：646分
- 636分位次：15066
- 查询结果：452条数据

### 注意事项

1. **降级方案**：如果score_rank_table查询失败，自动降级使用原有分数查询逻辑
2. **位次排序**：位次越小排名越高，查询范围使用[minRank, maxRank]
3. **科目类型映射**：
   - score_rank_table: "物理"、"历史"、"综合"
   - admission_plan: "物理"、"历史"
   - 前端传入: "物理,化学,生物"

### 后续优化建议

1. 添加索引优化score_rank_table查询性能：
   ```sql
   CREATE INDEX idx_province_subject_score ON score_rank_table(province, subject_type, score);
   CREATE INDEX idx_province_subject_rank ON score_rank_table(province, subject_type, cumulative_count);
   ```

2. 缓存热门省份的分数位次对照表，减少数据库查询

3. 前端显示位次信息，提升用户体验

## 修复浏览器跟踪防护导致的CDN加载失败

**修改时间**: 2026-04-09 11:40

### 问题描述

用户反馈在Edge浏览器中控制台出现错误：
```
Tracking Prevention blocked access to storage for <URL>
```

这是Edge浏览器的跟踪防护功能阻止了对CDN资源（cdn.jsdelivr.net）的访问，导致Bootstrap样式无法正常加载。

### 解决方案

将Bootstrap资源从CDN改为本地引用，完全绕过浏览器的跟踪防护机制。

### 修改内容

#### 1. 创建本地资源目录
```
e:\xm\vendor\bootstrap\
├── css\
│   └── bootstrap.min.css (232KB)
└── js\
    └── bootstrap.bundle.min.js (80KB)
```

#### 2. 修改HTML文件的资源引用

**修改文件**：
- index-mysql.html
- college-list.html
- recommend-results.html
- major-list.html
- school-detail.html

**修改内容**：
```html
<!-- 修改前 -->
<link href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css" rel="stylesheet">
<script src="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/js/bootstrap.bundle.min.js"></script>

<!-- 修改后 -->
<link href="vendor/bootstrap/css/bootstrap.min.css" rel="stylesheet">
<script src="vendor/bootstrap/js/bootstrap.bundle.min.js"></script>
```

#### 3. 资源下载

使用PowerShell命令下载Bootstrap资源：
```powershell
Invoke-WebRequest -Uri "https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css" -OutFile "e:\xm\vendor\bootstrap\css\bootstrap.min.css"
Invoke-WebRequest -Uri "https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/js/bootstrap.bundle.min.js" -OutFile "e:\xm\vendor\bootstrap\js\bootstrap.bundle.min.js"
```

### 优势

1. **稳定性提升**：不依赖外部CDN，避免网络波动影响
2. **兼容性强**：不受浏览器跟踪防护、广告拦截插件影响
3. **加载更快**：本地资源加载速度更快，无需DNS解析和跨域请求
4. **离线可用**：完全离线环境下也能正常工作
5. **版本固定**：避免CDN自动更新导致的兼容性问题

### 验证结果

- ✅ 所有HTML文件已更新为本地引用
- ✅ Bootstrap CSS文件已下载（232,914字节）
- ✅ Bootstrap JS文件已下载（80,421字节）
- ✅ 无linter错误

### 后续建议

1. 如果使用其他CDN资源（如jQuery、Font Awesome等），也建议改为本地引用
2. 考虑使用构建工具（如Vite、Webpack）自动管理本地依赖

## index-mysql.html 院校排序修复

**修改时间**: 2026-04-09 11:55

### 问题描述

用户反馈排序问题在 index-mysql.html 页面，而不是 recommend-results.html。

### 问题原因

index-mysql.html 的 renderTable 函数中，院校排序只按 college_level 排序，未按特色标签数量排序。

### 修改内容

**修改文件**: `e:\xm\index-mysql.html`

**修改位置**: 第2976-2987行

**修改前**：
```javascript
function getLvPrio(lv){...}
const groups=[...schoolMap.values()].sort((ga,gb)=>getLvPrio(ga[0].college_level||'')-getLvPrio(gb[0].college_level||''));
```

**修改后**：
```javascript
function getLvPrio(lv){...}
// 排序：先按特色标签数量降序，再按院校层次优先级
const groups=[...schoolMap.values()].sort((ga,gb)=>{
    const schoolNameA = ga[0].name || '';
    const schoolNameB = gb[0].name || '';
    const featureCountA = (collegeFeaturesMap[schoolNameA] || []).length;
    const featureCountB = (collegeFeaturesMap[schoolNameB] || []).length;
    // 先按特色标签数量降序
    if(featureCountA !== featureCountB) return featureCountB - featureCountA;
    // 数量相同，按院校层次优先级
    return getLvPrio(ga[0].college_level||'') - getLvPrio(gb[0].college_level||'');
});
```

### 排序规则

1. **第一优先级**：特色标签数量（降序）
   - 特色标签多的院校排在前面

2. **第二优先级**：院校层次（985 > 211 > 双一流 > 公办 > 民办）
   - 特色标签数量相同时，按院校层次排序

### 注意事项

index-mysql.html 中数据行使用 `row.name` 作为院校名称字段，与 recommend-results.html 使用的 `row.school_name` 不同。

```sql
CREATE TABLE major_introduction (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
    subject_category VARCHAR(50) COMMENT '学科门类',
    major_class VARCHAR(50) COMMENT '专业类',
    level VARCHAR(20) COMMENT '层次',
    major_name VARCHAR(100) COMMENT '专业名称',
    major_code VARCHAR(20) COMMENT '专业代码',
    study_years VARCHAR(20) COMMENT '修业年限',
    degree VARCHAR(50) COMMENT '授予学位',

    major_intro TEXT COMMENT '专业是什么',
    major_content TEXT COMMENT '专业学什么',
    career_direction TEXT COMMENT '专业干什么',
    employment_destination TEXT COMMENT '就业去向',
    employment_region TEXT COMMENT '就业地区分布',
    employment_industry TEXT COMMENT '就业行业分布',
    employment_position TEXT COMMENT '就业岗位分布',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_major_name (major_name),
    INDEX idx_major_code (major_code),
    INDEX idx_subject_category (subject_category),
```
